⚡
SwiftSignalKit
响应式编程框架 · 整个 app 的神经系统
▸

一句话解释：一个让数据能「自动流动」的工具。当数据变了，所有关心这个数据的地方都会自动收到通知并更新。

你关注了一个博主，博主发新动态你就会收到推送——不需要你每隔五分钟自己去刷新。SwiftSignalKit 就是这个「关注 + 推送」机制，只不过是给代码用的。

Telegram 自己写了这个框架（比苹果的 Combine、社区的 RxSwift 都早），整个 app 几乎所有模块都依赖它。核心概念只有两个：

Signal — 一个数据流。比如「聊天列表的最新状态」就是一个 Signal，它会在有新消息时自动发出新的值。

Promise — 一个可以被手动更新的 Signal。当你想「主动往数据流里塞一个新值」的时候用它。

为什么重要？因为 Telegram 的 UI 从来不会「主动去问数据有没有变」——它只是坐在那等着。数据一变，Signal 自动推过来，UI 自动更新。这就是「响应式」。
Starlet 的群聊如果也采用响应式数据流，消息到达时 UI 可以自动更新，不需要手动刷新或者轮询。这也是 Telegram 滑动如丝的原因之一。
💾
Postbox — 本地真源层
加密本地数据库 · 所有数据的唯一来源
▸

一句话解释：手机上的一个加密数据库，存着你所有的聊天记录、联系人、媒体文件信息。UI 显示的所有数据都从这里读，不从网络直接读。

你每天看的不是新闻直播（网络），而是自己手机里存好的新闻 app 缓存（Postbox）。新闻一到，先存进手机，然后你从手机上看。这样即使断网了，之前的新闻还都在。

Postbox 内部分成两个角色：

Table（表）— 负责写入。新消息来了，就往 Table 里写。

View（视图）— 负责读取 + 自动通知。UI 订阅一个 View，Table 一有变化，View 就会通过 SwiftSignalKit 的 Signal 推送给 UI。

这就是 Cele 提到的「本地真源层」。所有数据的流动路径是：

服务器 → 写入 Postbox → Postbox 通知 UI → UI 更新

UI 永远不直接问服务器要数据。这保证了离线可用、滑动流畅、数据一致。

底层用的是 SQLCipher（加密版的 SQLite），所以即使有人拿到你的手机文件系统，没有密钥也读不了聊天记录。

Starlet 现在的 H5 群聊可能还是直接从服务器拉数据——如果改成「先存本地、从本地读」的模式，离线体验和滚动流畅度都会大幅提升。原生重写的时候可以参考这个思路。
🌐
MtProtoKit + TelegramApi
网络通信层 · 自研加密协议
▸

一句话解释：负责跟 Telegram 服务器对话的模块。用的不是普通的 HTTPS，而是 Telegram 自己发明的加密通信协议 MTProto 2.0。

如果说普通 app 跟服务器说话像写明信片（HTTPS），Telegram 就是用密码本写暗号信（MTProto）。连邮差（网络运营商）都看不懂信件内容。

这层分两个部分：

MtProtoKit — 处理连接、加密、选择最近的数据中心、断线重连。你不需要理解加密细节，只要知道它把「跟服务器安全通信」这件事包好了。

TelegramApi — 自动生成的代码，把服务器支持的几百个 API 接口变成 Swift 可以直接调用的函数。比如「发送消息」「获取聊天列表」。

TelegramApi 的代码是机器生成的，不是人手写的——它是从 Telegram 的 API 规范文件（叫 TL Schema）自动转换过来的。所以这部分代码不需要读，看不懂是正常的。
⚙️
TelegramCore
业务逻辑中枢 · 串联数据库和网络
▸

一句话解释：整个 app 的「大脑」，负责处理所有业务逻辑——发消息、同步聊天、管理设置、处理通知。它从 Postbox 读数据，通过 MtProtoKit 发请求，再把结果写回 Postbox。

回到餐厅的比喻：大厨（TelegramCore）从冰箱（Postbox）拿食材，有时候让外卖骑手（MtProtoKit）去进货，做好菜放回冰箱，服务员（UI）从冰箱取菜端给客人。大厨是整个流程的协调者。

核心入口是 TelegramEngine——它是 TelegramCore 对外的统一接口。UI 层的代码不会直接去碰 Postbox 或 MtProtoKit，而是通过 TelegramEngine 来操作一切。

比如「发送一条消息」的完整流程是：

流程示意（不是真实代码）
// 1. UI 调用 TelegramEngine 发消息 engine.sendMessage(to: chatId, text: "你好") // 2. TelegramCore 先把消息存进 Postbox（状态：发送中） // → UI 立刻看到消息出现（带个小时钟图标） // 3. TelegramCore 通过 MtProtoKit 发给服务器 // 4. 服务器确认收到 // → TelegramCore 更新 Postbox 里的消息状态（已发送） // → UI 自动更新（小时钟变成对勾 ✓）
注意第 2 步：消息还没真的发出去，UI 上就已经显示了。这就是为什么 Telegram 发消息感觉瞬间完成——它先在本地「假装」发成功了（乐观更新），然后后台慢慢跟服务器确认。
Starlet 的消息发送如果也用这种「乐观更新」策略——先本地展示、后台确认——体感速度会快很多，用户不用盯着转圈等服务器响应。
🔑
AccountContext
依赖注入中心 · 每个页面的万能钥匙
▸

一句话解释：一个「百宝箱」对象，传给 app 里的每一个页面，里面装着当前用户的所有服务——数据库、网络、设置、主题等等。

你去酒店入住，前台给你一张房卡。这张卡能开房门、刷健身房、刷泳池、刷餐厅——一张卡通行所有设施。AccountContext 就是 app 里每个页面的「房卡」。

为什么这么设计？因为如果每个页面都要自己去找数据库在哪、网络在哪、当前用户是谁，代码会乱成一团。有了 AccountContext，只要拿着它就能访问一切，而且还能支持多账号切换——换一个 AccountContext 就是换一个账号。

这种设计模式叫「依赖注入」——不让页面自己去找它需要的东西，而是在创建页面的时候就把东西递给它。好处：页面不需要知道东西从哪来，只管用就行。
🎨
Display + AsyncDisplayKit + ComponentFlow
自研 UI 框架 · Telegram 丝滑的秘密
▸

一句话解释：Telegram 没有用苹果自带的 UIKit 或 SwiftUI 来画界面，而是自己搭了一套 UI 系统。这就是为什么它滑起来比大多数 app 流畅。

苹果给的 UIKit 就像宜家家具——能用，但大家的房子都长差不多。Telegram 选择自己做定制家具（AsyncDisplayKit），用更复杂的工艺换来更好的效果。

这套系统分三层：

AsyncDisplayKit（底层）
原本是 Facebook 开源的框架（也叫 Texture），Telegram fork 了一份自己维护。核心能力：把 UI 的计算和绘制放到后台线程做，主线程只负责最后一步「把画好的内容贴到屏幕上」。这就是为什么你在 Telegram 里快速滑动聊天列表不会卡顿——它在你滑之前就已经在后台把下面的内容画好了。

Display（中间层）
在 AsyncDisplayKit 上面封装了一些常用的 UI 组件：导航栏、列表、弹窗、页面容器等。

ComponentFlow（最新层）
Telegram 自己做的「声明式 UI」框架，类似苹果的 SwiftUI 但更早。新功能（Stories、Emoji 键盘等）都用 ComponentFlow 来写。

Telegram 有一个独特的 UI 规则：一个 view 永远不设置自己的大小和位置，只有它的父级才能设置。这条规则贯穿整个 app，保证了布局的一致性和可预测性。
Starlet 重写原生群聊时，核心收益点在这里：用 AsyncDisplayKit 的思路，把消息气泡的布局和渲染放到后台线程，主线程只做「展示」。这就是 H5 永远比不上原生的地方——H5 的 JavaScript 是单线程的，没法做到这一点。
🖥
TelegramUI
用户界面 · 200+ 个 UI 组件
▸

一句话解释：你在 Telegram 里看到的所有页面——聊天界面、设置页、贴纸面板、Stories 浏览器、通话界面等等——都在这个模块里。

TelegramUI 内部有 200 多个子组件，全放在 TelegramUI/Components/ 目录下。几个跟 Starlet 最相关的：

ChatController — 聊天页面的主控制器，是整个 app 最大的文件之一，被拆成了很多 extension 文件来管理。

ChatHistoryListNode — 聊天消息列表，负责消息的滚动和展示。

ChatMessageBubbleItemNode — 单条消息气泡的渲染。

TelegramUI 本身没什么"高深"设计，它更像是「在上面几层提供的能力上组装具体页面」。真正的架构智慧在下面几层——SwiftSignalKit、Postbox、Display。读源码时，这层是最后看的，因为不理解下面几层就看不懂这层在干嘛。
Focus Area

群聊消息系统是怎么跑的

既然 Starlet 要重写群聊，这部分值得单独拿出来看。

💬
消息从收到到显示的完整流程
5 步走完一条消息的一生
▸
一条新消息的旅程
① 服务器推送 服务器通过 MTProto 连接推送一条新消息 ② TelegramCore 接收并处理 解析消息内容、发送者信息、时间戳等 ③ 写入 Postbox 存进 MessageHistoryTable（消息历史表） 同时更新 ChatListTable（聊天列表排序） ④ Postbox View 感知到变化 MessageHistoryView 检测到新数据 通过 SwiftSignalKit 的 Signal 发出通知 ⑤ UI 自动更新 ChatHistoryListNode 收到 Signal 渲染新的消息气泡 聊天列表也自动更新排序和预览文本

整个过程 UI 层没有做任何主动查询——它只是被动接收 Signal 推来的数据变化。这就是响应式架构的威力。

Starlet 群聊重写的核心参考路径：

1. 建一个本地消息数据库（类似 Postbox），用 SQLite 就行
2. 消息到达时先存本地，再通知 UI
3. UI 订阅数据变化，不要自己轮询
4. 消息列表的渲染放到后台线程（参考 AsyncDisplayKit 思路）
5. 发送消息用乐观更新——先本地展示，后台确认

Fun Facts

读源码时发现的有趣细节

这些可能会让你对 Telegram 的工程能力更有感觉。

🔒 零测试
整个项目没有一个单元测试。听起来很疯狂，但 Telegram 的质量靠的是极度严格的分层架构和超长的内测期（alpha → beta → stable 三轮），而不是自动化测试。
🏗 构建系统用 Bazel
不用 Xcode 的构建系统，用 Google 开源的 Bazel。因为 250+ 模块如果用 Xcode 编译会慢到怀疑人生，Bazel 可以精确跟踪依赖、只重新编译改了的部分。
🎨 200+ UI 组件全手写
聊天气泡、emoji 键盘、贴纸面板、视频编辑器、相机界面、Stories 浏览器……全是自己从零搭的。一个组件一个文件夹，结构极清晰。
