Why

先搞清楚它为什么存在

在讲 SwiftSignalKit 怎么工作之前，先搞懂一个问题：没有它的时候，代码是怎么写的？为什么那样不行？

🤯
没有响应式的世界
你手动刷新过网页等外卖进度吗？
▸
你点了外卖，想知道骑手到哪了。有两种方式：

方式 A（轮询）：每隔 30 秒自己打开 app 看一眼。骑手没动？白刷了。骑手到了？可能已经等了 29 秒你才知道。

方式 B（推送）：骑手每到一个路口，app 自动给你弹一条通知。你不需要做任何事，信息自己来找你。

传统的编程方式就是方式 A——UI 需要数据的时候，主动去问「嘿，有没有新消息？有没有新数据？」这叫轮询，又笨又费电。

SwiftSignalKit 让 Telegram 用方式 B——数据变了，自动通知所有关心它的人。UI 只管坐着等，数据自己推过来。

响应式编程的核心思想就一句话：数据变了 → 自动通知 → UI 自动更新。不需要手动刷新，不需要轮询，不需要回调地狱。

Telegram 在 2015 年左右就自己写了 SwiftSignalKit。那时候苹果的 Combine 还没出生（2019 年才发布），RxSwift 也还不成熟。Telegram 的团队等不及别人，自己造了一个。

Core Components

四个核心零件

SwiftSignalKit 的所有文件加起来不到 3000 行代码。核心概念只有四个，搞懂它们就搞懂了整个框架。

⚡
Signal — 数据流
一切的起点，所有数据都通过它流动
▸

一句话：Signal 就是一根水管，水（数据）从一头流进去，另一头自动流出来给所有接上管子的人。

想象你在 B 站关注了一个 UP 主。

UP 主 = Signal 的数据源（generator）
发新视频 = Signal 发出一个新值（putNext）
你的关注列表自动出现新视频 = 你订阅了这个 Signal
UP 主停更了 = Signal 完成（putCompletion）
UP 主被封号了 = Signal 报错（putError）

Signal 有两个泛型参数：Signal<T, E>，其中 T 是数据类型，E 是错误类型。如果你确定不会出错，E 就写 NoError。

创建一个 Signal 很简单——给它一个 generator 闭包，告诉它「数据怎么来」：

Swift 伪代码
// 创建一个 Signal：倒计时 3 秒 let countdown = Signal<Int, NoError> { subscriber in subscriber.putNext(3) // 发出 3 subscriber.putNext(2) // 发出 2 subscriber.putNext(1) // 发出 1 subscriber.putCompletion() // 完成 return EmptyDisposable } // 订阅这个 Signal countdown.start(next: { value in print("倒计时: \(value)") }, completed: { print("发射！") }) // 输出： // 倒计时: 3 // 倒计时: 2 // 倒计时: 1 // 发射！
Signal 的 generator 只在有人 start（订阅）的时候才会执行。没人听的广播不会浪费电。这叫懒执行——你不 start，它就不动。

在 Telegram 里，Signal 无处不在。几个真实例子：

• 聊天列表的最新状态 → Signal<[Chat], NoError>
• 一条消息的发送进度 → Signal<MessageStatus, NoError>
• 用户的在线状态 → Signal<UserPresence, NoError>
• 网络连接状态 → Signal<ConnectionStatus, NoError>

📬
Subscriber — 收件人
接收数据的那一端
▸

一句话：Subscriber 就是快递柜——Signal 发出的数据会投递到 Subscriber，然后 Subscriber 通知你来取。

你网购了三个包裹。快递柜（Subscriber）的工作：

• 包裹到了 → 给你发取件码（next）
• 包裹丢了 → 通知你异常（error）
• 所有包裹都到齐了 → 通知你「全部收到」（completed）

一旦通知了异常或全部完成，快递柜就不再接收新包裹了。

Subscriber 内部有一把互斥锁（pthread_mutex），保证多个线程同时往里塞数据的时候不会乱套。这就是为什么 Telegram 可以在后台线程产生数据、在主线程更新 UI，中间不会崩溃。

Subscriber 的三个核心方法
putNext(value) // 投递一个新值，可以调用无数次 putError(error) // 投递一个错误，调用后 Subscriber 终止 putCompletion() // 标记完成，调用后 Subscriber 终止 // ⚠️ 一旦调用了 putError 或 putCompletion // 后续的 putNext 会被静默忽略
你平时不需要直接创建 Subscriber——它是 Signal 内部自动管理的。你只需要在 .start(next:error:completed:) 里写三个闭包，Subscriber 会帮你把数据分发到对应的闭包。
🗑
Disposable — 退订机制
关掉水龙头，别让水白流
▸

一句话：Disposable 就是取消订阅的按钮。你不想听了，按一下，Signal 就不再给你发数据了。

你订了三个杂志。搬家的时候需要挨个打电话退订，不然杂志会一直寄到旧地址浪费钱。

Disposable 就是那个「退订电话」。不调用的话，Signal 会一直在后台跑，白白消耗内存和 CPU。

SwiftSignalKit 提供了四种 Disposable，覆盖不同场景：

类型 作用 生活类比
ActionDisposable 退订时执行一个自定义动作 退订杂志时顺便发个投诉邮件
MetaDisposable 可以替换内容——设新的会自动取消旧的 换一家杂志订阅，自动帮你退掉上一家
DisposableSet 一组订阅打包管理，一键全退 搬家时一个电话退掉所有杂志
DisposableDict 按名字管理一组订阅，可以单独退某一个 记着每家杂志的名字，想退哪家退哪家
最常见的用法
class ChatController { // 用 MetaDisposable 管理订阅 private let messageDisposable = MetaDisposable() func loadMessages() { // 每次调用都会自动取消上一次的订阅 messageDisposable.set( fetchMessages().start(next: { messages in self.updateUI(messages) }) ) } deinit { // 页面销毁时，退订所有还在跑的任务 messageDisposable.dispose() } }
如果你忘了 dispose，Signal 就会一直在后台跑，即使那个页面已经关掉了。这是内存泄漏最常见的原因之一。Telegram 的做法是：在每个控制器的 deinit 里统一清理所有 Disposable。
🔄
Promise & ValuePromise — 可变数据源
当你需要手动更新数据的时候
▸

一句话：Signal 是水管，那 Promise 就是水龙头——你可以随时拧开它，往水管里灌新的水。

普通的 Signal 一旦创建，数据源就定死了（由 generator 决定）。但很多时候你需要动态更新数据——比如用户输入了新的搜索关键词，你需要把新的关键词推进数据流。这就是 Promise 干的事。

Signal = 电视直播，你只能看台里播什么。
Promise = 你自己的投影仪，你想播什么就播什么，还能随时换片。
ValuePromise = 一块白板，你擦掉旧内容写新的，所有看着白板的人立刻看到变化。

Promise 和 ValuePromise 的区别：

Promise<T> ValuePromise<T>
更新方式 用另一个 Signal 来更新
.set(signal)		 直接塞一个值
.set(value)
适用场景 数据来自异步操作（网络请求、数据库查询） 数据来自同步操作（用户输入、状态切换）
去重 不去重，每次 set 都通知 值没变就不通知（需要 T 实现 Equatable）
Promise vs ValuePromise
// Promise：用 Signal 更新 let chatList = Promise<[Chat]>() chatList.set(fetchChatsFromDatabase()) // 传入一个异步 Signal // ValuePromise：直接塞值 let searchText = ValuePromise<String>("") searchText.set("hello") // 用户输入了 hello，直接更新 searchText.set("hello") // 重复值，ValuePromise 不会重复通知 searchText.set("world") // 新值，通知所有订阅者
Starlet 的群聊输入框可以用 ValuePromise 来管理「正在输入」状态——用户打字时 set(true)，停下来 set(false)，ValuePromise 自动去重，不会因为用户每打一个字就发一次通知。
Data Flow

数据怎么流动的

知道了四个零件，接下来看它们怎么拼在一起。

🌊
一条完整的数据流
从产生到消费的全过程
▸
想象一条河：

源头（Signal 的 generator）→ 河道（操作符加工数据）→ 分流口（多个订阅者）→ 水闸（Disposable 控制开关）
📡 数据源 Signal generator
↓ putNext
🔧 操作符加工 map / filter / delay
↓ 变换后的数据
📬 Subscriber 接收并分发
↓ 回调
🖥 UI 更新 next 闭包里刷新界面

用 Telegram 里的一个真实场景来走一遍：搜索联系人。

搜索联系人的数据流
// 1. 用户在搜索框打字 → ValuePromise 收到新的搜索词 let searchQuery = ValuePromise<String>("") // 2. 搜索词通过一系列操作符加工 let results = searchQuery.get() // 拿到 Signal |> filter { $0.count >= 2 } // 少于 2 个字不搜 |> delay(0.3, queue: .main) // 等 0.3 秒（防抖） |> mapToSignal { query in // 发起搜索请求 searchContacts(query: query) } // 3. UI 订阅结果 disposable.set( results.start(next: { contacts in self.showResults(contacts) // 更新搜索结果列表 }) )

这段代码做了什么？

用户打字 → 不到 2 个字的忽略 → 打完等 0.3 秒才搜（不是每打一个字就搜一次）→ 发起搜索 → 结果自动推到 UI。

整个过程没有一个 if-else，没有一个定时器，没有一个回调嵌套。数据像水一样流过每一个环节，每个环节只管自己那一步。

注意那个 |> 符号——这是 SwiftSignalKit 的「管道操作符」，借鉴了 JavaScript 的 pipeline operator 提案。它让代码的阅读顺序跟数据的流动方向一致：从上往下读就是从源头到终点。
Operators

常用操作符

操作符就是数据流中间的加工站。原始数据流过去，出来的时候已经变成你想要的样子了。

🔧
变换类：map / filter / mapToSignal
改形状、筛选、展开嵌套
▸
流水线上的工序：

map = 喷漆车间（进来白色的，出去红色的，形状不变）
filter = 质检站（合格的放行，不合格的扔掉）
mapToSignal = 拆箱站（进来一个大箱子，拆开后里面的小东西一个一个放上传送带）
三个最常用的操作符
// map — 把一种数据变成另一种 // 例：把用户对象变成用户名字符串 userSignal |> map { user in user.name } // Signal<User> → Signal<String> // filter — 只放行满足条件的值 // 例：只要在线的用户 userSignal |> filter { user in user.isOnline } // Signal<User> → Signal<User>（但只有在线的才会通过） // mapToSignal — 把值变成另一个 Signal，然后展平 // 例：拿到用户 ID 后去请求用户详情 userIdSignal |> mapToSignal { id in fetchUserDetails(id: id) } // Signal<Int> → Signal<UserDetails>

mapToSignal 是最容易让人迷糊的。简单理解：map 是同步变换（拿到值立刻变），mapToSignal 是异步变换（拿到值之后要等一个异步操作完成才能拿到结果）。

在 Telegram 的代码里，mapToSignal 出现的频率极高——因为大量操作是异步的：查数据库、请求网络、等用户确认。每次遇到「拿到 A 之后去请求 B」的场景，就是 mapToSignal。
🔀
组合类：combineLatest / then
多条数据流汇合
▸
combineLatest：你同时等两个外卖——只要其中一个有更新（骑手位置变了），你就重新看一眼两个外卖的最新状态。

then：吃完午饭再吃甜点。午饭（第一个 Signal）完成了，才开始甜点（第二个 Signal）。
combineLatest
// 聊天页面需要同时知道：消息列表 + 用户在线状态 // 任何一个变了，都要重新渲染 let combined = combineLatest( messagesSignal, // Signal<[Message]> presenceSignal // Signal<UserPresence> ) // 结果：Signal<([Message], UserPresence)> // 消息更新了 → 触发，拿到最新消息 + 最新在线状态 // 在线状态变了 → 触发，拿到最新消息 + 最新在线状态
combineLatest 在 Telegram 里的使用密度极高。一个聊天页面要同时监听消息、用户状态、网络状态、主题设置等七八个数据源——全靠 combineLatest 把它们组合在一起。
⏱
时间类：delay / timeout
控制数据流动的节奏
▸
delay：电梯关门等 3 秒。不是电梯坏了，是故意等一等看有没有人要上来。

timeout：外卖超过 30 分钟没到就自动退单。

搜索防抖是 delay 最经典的用法——用户打字很快，你不想每打一个字母就发一次网络请求。加一个 delay(0.3)，用户停下来 0.3 秒才真正搜索。如果 0.3 秒内又打了新字，旧的请求自动取消。

防抖原理
// 用户快速输入 "h" "he" "hel" "hell" "hello" // 没有 delay：发 5 次请求，浪费 4 次 // 有 delay(0.3)：只在用户停顿后发 1 次请求 "hello" searchQuery.get() |> delay(0.3, queue: .mainQueue()) |> mapToSignal { query in searchAPI(query) }
Starlet 的搜索功能如果还没做防抖，加一个就行。不加的话，用户打字快的时候服务器会收到一堆无意义的请求，浪费带宽还可能触发限频。
🧵
线程类：deliverOnMainQueue
确保 UI 更新在主线程
▸

iOS 有一条铁律：所有 UI 操作必须在主线程执行。但数据往往是在后台线程产生的（网络请求返回、数据库查询完成）。

你在工厂（后台线程）生产零件，但安装必须在展厅（主线程）进行。deliverOnMainQueue 就是从工厂到展厅的传送带——零件在工厂做好，自动送到展厅去安装。
线程切换
// 数据库查询在后台线程执行 // deliverOnMainQueue 把结果切到主线程 // next 闭包里可以安全地更新 UI fetchFromDatabase() |> deliverOnMainQueue |> start(next: { data in self.tableView.reloadData() // 安全！在主线程 })

SwiftSignalKit 的 Queue 类包装了 GCD，提供了三个预设队列：

• Queue.mainQueue() — 主线程，更新 UI 用
• Queue.concurrentDefaultQueue() — 默认并发队列，一般计算用
• Queue.concurrentBackgroundQueue() — 后台队列，重活用

Real World

在 Telegram 里的真实用法

理论讲完了，来看两个从 Telegram 源码里抽出来的真实场景。

💬
场景一：用户名实时校验
打字 → 防抖 → 本地校验 → 服务端校验 → 显示结果
▸

设置用户名的时候，Telegram 会实时告诉你这个名字能不能用。这个流程全靠 SwiftSignalKit 串起来：

⌨️ 用户打字 ValuePromise 更新
↓
⏱ delay 0.3s 防抖，等用户停下来
↓
✅ 本地校验 长度、字符合法性
↓ 通过了才继续
🌐 服务端校验 检查是否已被占用
↓
🖥 显示结果 ✓ 可用 / ✗ 已被占用

关键细节：用的是 MetaDisposable——用户每次打新字，旧的服务端校验请求自动取消。不会出现「查 abc 的结果比查 ab 的结果晚到，覆盖了正确结果」的 bug。

这就是响应式编程真正的价值：用声明式的方式描述「数据怎么流、怎么变」，而不是用一堆 if-else、flag、callback 来手动管理状态。代码量少了一半，bug 也少了一半。
📱
场景二：聊天页面的数据编排
七八个数据流同时跑，一个 combineLatest 搞定
▸

打开一个聊天页面，你看到的不只是消息列表——还有对方的在线状态、头像、输入中提示、背景主题、网络状态指示器……这些数据来自不同的 Signal，但需要一起渲染。

聊天页面的数据组合（简化）
let chatData = combineLatest( messagesSignal, // 消息列表 peerPresenceSignal, // 对方在线状态 typingStatusSignal, // "对方正在输入..." themeSignal, // 当前主题/壁纸 networkStatusSignal // 网络连接状态 ) |> deliverOnMainQueue chatDataDisposable.set( chatData.start(next: { messages, presence, typing, theme, network in // 任何一个数据源变了，这里都会触发 // 拿到的永远是所有数据源的最新值 self.render( messages: messages, isOnline: presence.isOnline, isTyping: typing, theme: theme, isConnected: network.isConnected ) }) )

如果不用响应式，你需要：给每个数据源写一个回调 → 每个回调里手动检查其他数据源的状态 → 手动决定是否需要刷新 UI → 处理各种时序问题（A 的回调先到还是 B 的先到）。

用 combineLatest，所有这些问题都没了。任何一个变，拿到所有的最新值，刷一次 UI。就这么简单。

Comparison

跟别的响应式框架比

Combine、RxSwift 你可能听过。SwiftSignalKit 跟它们有什么区别？

⚖️
SwiftSignalKit vs Combine vs RxSwift
三个框架各有什么特点
▸
SwiftSignalKit Combine RxSwift
谁做的 Telegram 团队 Apple 社区
诞生时间 ~2015 2019 ~2016
核心类 Signal Publisher Observable
代码量 ~3000 行 闭源 ~20000 行
独特设计 |> 管道操作符 链式调用 链式调用
学习曲线 低（概念少） 中 高（概念多）
SwiftSignalKit 的哲学是够用就好。它没有 RxSwift 的几百个操作符，没有 Combine 的 async/await 集成——但 Telegram 用它撑起了一个日活 9 亿的 app。核心概念少 = 学得快、出 bug 少、维护简单。

Telegram 为什么不换成 Combine？因为 SwiftSignalKit 已经渗透到每一行代码里了，换掉等于重写整个 app。而且 Combine 要求 iOS 13+，Telegram 当年还要支持更老的系统。到现在，已经没有换的必要了——它跑得好好的。

Cheat Sheet

一张图记住 SwiftSignalKit

以后看 Telegram 源码遇到不认识的东西，翻回来查这里。

Signal<T, E>
数据流。T 是数据类型，E 是错误类型。generator 里定义数据怎么来。
Subscriber
接收端。putNext 发数据，putError 报错，putCompletion 完成。线程安全。
Disposable
退订。MetaDisposable 最常用——set 新的自动取消旧的。deinit 里必须 dispose。
Promise / ValuePromise
可变数据源。Promise 用 Signal 更新，ValuePromise 直接塞值且自动去重。
常用操作符：map filter mapToSignal combineLatest delay deliverOnMainQueue
